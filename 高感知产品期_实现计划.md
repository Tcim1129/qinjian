# 亲健高感知产品期实现计划

## 目标

把三项高感知能力拆成真正可开发的顺序：

1. 聊天前仿真
2. 双视角叙事对齐
3. 冲突修复协议

## 第一批：先做接口与数据结构

### 1. 聊天前仿真

- 定义请求结构：原句、关系上下文、可选目标对象
- 定义返回结构：风险评分、第一感受、改写建议、建议/避免项
- 定义事件名：`message.simulated`
- 定义写回字段：风险变化、推荐动作、是否采纳建议

### 2. 双视角叙事对齐

- 定义请求结构：pair_id、时间范围、记录来源
- 定义返回结构：共同版本、双方关注点、错位点、桥接句
- 定义事件名：`alignment.generated`
- 定义写回字段：错位类型、桥接效果、后续沟通结果

### 3. 冲突修复协议

- 定义请求结构：风险等级、触发事件、关系画像快照
- 定义返回结构：协议等级、步骤列表、暂停条件、复盘建议
- 定义事件名：`repair_protocol.requested`
- 定义写回字段：协议执行状态、完成率、后续效果

## 第二批：后端服务落地

### 1. 新增或补齐服务层

- 消息仿真服务
- 叙事对齐服务
- 修复协议编排服务

### 2. 接入核心引擎

- 所有服务统一读取关系画像
- 所有结果统一回写事件流
- 所有执行结果统一进入效果回流

### 3. API 层要求

- 返回结构稳定
- 失败场景有兜底文本
- 单人模式与双人模式要提前定义差异

## 第三批：Web 先做完整主链路

### 1. 聊天前仿真页 / 面板

- 输入一句准备发的话
- 立即看到风险与改写建议
- 一键复制更安全版本

### 2. 双视角叙事对齐页 / 面板

- 展示共同版本
- 展示双方更在意的点
- 展示桥接句

### 3. 修复协议页 / 面板

- 展示协议等级
- 用步骤卡片推进执行
- 支持“已完成 / 暂停 / 稍后继续”

## 第四批：回流与评估

### 要记录的数据

- 用户是否用了改写建议
- 用户是否打开了桥接句
- 用户是否执行了协议步骤
- 执行后下一次打卡和风险状态是否改善

### 最小评估指标

- 消息仿真建议采纳率
- 双视角对齐打开后沟通继续率
- 修复协议完成率
- 执行后 3 天 / 7 天风险下降率

## 推荐开发顺序

1. 先做聊天前仿真
2. 再做双视角叙事对齐
3. 最后做冲突修复协议完整执行态

原因：

- 聊天前仿真最轻，最容易先形成高感知价值
- 双视角叙事对齐能体现“这产品懂双方”
- 修复协议最重，适合在前两者结构稳定后再补完整

## 完成标准

这份实现计划完成，不是文档写完，而是达到：

1. 三项能力都有明确 API 和数据结构
2. Web 已形成完整演示链路
3. 回流指标已能开始积累
