# 亲健本地测试与上云检查清单

本文档的目标很直接：先在本地把关键链路跑通，再判断是否已经达到可上云状态。

---

## 一、先决条件

本地建议环境：

- Windows 10/11 或 Ubuntu 22.04+
- Python 3.12
- PostgreSQL 16
- Docker Desktop（如果你走容器化测试）
- Node.js 不是必须，当前 Web 前端是原生 HTML/CSS/JS

你至少需要准备以下内容：

- 一个可用的 SiliconFlow API Key
- 一个本地 PostgreSQL 数据库
- 仓库代码已拉到本机

---

## 二、本地测试方式选择

你可以选两种方式。

### 方式 A：先跑最小可用联调

适合先验证功能，不先折腾 Docker。

- 后端本机运行
- PostgreSQL 本机运行
- Web 前端用浏览器或静态服务器打开

### 方式 B：直接按部署形态测试

适合你想尽快验证“这套东西能不能上云”。

- PostgreSQL、后端、Nginx 全部走 Docker Compose
- 本地先按准生产形态验证，再上服务器

如果你当前目标是“可以就上云”，建议先走方式 A 跑通功能，再走方式 B 做上线前检查。

---

## 三、方式 A：本机联调步骤

### 1. 创建数据库

先确保 PostgreSQL 已启动，然后创建数据库和用户。

```sql
CREATE USER qinjian WITH PASSWORD 'qinjian';
CREATE DATABASE qinjian OWNER qinjian;
```

如果你已经有现成数据库，也可以直接复用，只要后面的连接串一致。

### 2. 配置后端环境变量

在 [backend/.env.example](backend/.env.example) 的基础上复制一份为 `backend/.env`，至少填写这些值：

```env
SECRET_KEY=change-this-to-a-long-random-string
DATABASE_URL=postgresql+asyncpg://qinjian:qinjian@localhost:5432/qinjian
SILICONFLOW_API_KEY=你的真实APIKey
SILICONFLOW_BASE_URL=https://api.siliconflow.cn/v1
AI_MULTIMODAL_MODEL=moonshot/kimi-k2.5
AI_TEXT_MODEL=deepseek-ai/DeepSeek-V3
```

如果你要本地调试手机号验证码，可以额外临时加：

```env
DEBUG=true
PHONE_CODE_DEBUG_RETURN=true
```

这个配置只建议本地测试时使用。准备上云前，把 `PHONE_CODE_DEBUG_RETURN` 改回 `false`。

### 3. 安装后端依赖

在项目根目录执行：

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### 4. 执行数据库迁移

如果是首次本地联调，或者你之前已经跑过旧版本数据库，建议先执行一次 Alembic 迁移：

```powershell
cd backend
.\.venv\Scripts\Activate.ps1
alembic upgrade head
```

说明：

- 当前后端启动时虽然保留了 `create_all` 兜底，但它不会补齐已存在表的结构变更；
- 仓库里已经有 Alembic 版本文件，联调前执行 `alembic upgrade head` 更稳。

### 5. 启动后端

```powershell
cd backend
.\.venv\Scripts\Activate.ps1
uvicorn app.main:app --reload --port 8000
```

启动后先访问：

- http://localhost:8000/api/health

看到返回 `{"status":"ok"...}` 说明后端基础可用。

### 6. 打开前端

新开一个终端，在项目根目录执行：

```powershell
cd web
python -m http.server 3000
```

然后浏览器访问：

- http://localhost:3000

当前 [web/js/api.js](web/js/api.js) 会在本地自动把 API 根地址指向 `http://localhost:8000/api/v1`，所以不需要你额外改前端配置。

## 四、本地功能测试顺序

建议按这个顺序测，能最快定位问题。

### 1. 基础健康检查

- 打开首页，确认页面能正常加载
- 确认登录页 UI 正常，无控制台报错
- 访问后端健康检查地址，确认 API 正常

### 2. 账号与认证

- 测试邮箱注册
- 测试邮箱登录
- 测试手机号验证码发送
- 测试手机号验证码登录
- 测试退出登录
- 测试刷新页面后 token 是否保持

如果要测手机号验证码：

- 先调用发送验证码
- 再用返回的 `debug_code` 登录
- 验证未注册手机号首次登录时是否自动创建账号
- 测试验证码过期、重发冷却、输错次数限制是否生效

### 3. 配对主链路

- 创建一个关系
- 记录生成的邀请码
- 用第二个账号加入配对
- 验证首页是否出现当前关系摘要
- 验证多关系切换是否正常

### 4. 打卡与报告

- 用一方提交打卡，确认生成单方日记或状态变化
- 用双方都提交打卡，确认日报可生成
- 测试周报、月报生成条件是否符合预期
- 验证趋势图、历史记录、危机预警能否正常展示

### 5. 里程碑与个人中心

- 新建里程碑
- 查看首页里程碑摘要是否刷新
- 进入里程碑页生成成长回顾
- 在个人中心修改备注名，确认首页和个人中心同步刷新

### 6. 任务、依恋、异地功能

- 触发依恋分析
- 查看每日关系任务
- 创建异地活动并完成
- 查看异地健康指数

## 五、网页端自动回归

仓库已经带了一个现成脚本 [test_ui.py](test_ui.py)，这个脚本不依赖后端，主要用来检查 Web 工作台的页面结构和基础交互有没有坏。

运行方式：

```powershell
python test_ui.py
```

通过标准：

- 所有页面截图都成功输出到 `screenshots/`
- 控制台错误/警告为 0
- Discover 页面能看到“关系里程碑”入口
- 健康测试全流程完成

如果这一步不过，不建议直接上云。

## 六、方式 B：本地按容器形态测试

如果你想用接近生产环境的方式验证，直接在项目根目录执行：

```powershell
docker compose up -d --build
```

然后检查：

```powershell
docker compose ps
docker compose logs -f backend
docker compose logs -f web
```

说明：

- 当前 [backend/Dockerfile](backend/Dockerfile) 默认通过 [backend/start.sh](backend/start.sh) 启动；
- 启动脚本会先执行 `alembic upgrade head`，迁移失败时才回退到 `create_all` 兜底逻辑；
- 所以 Docker 方式更接近你未来上云时的真实行为。

访问：

- http://localhost:8080
- http://localhost:8080/api/health

如果你使用 `docker-compose.yml` 的默认配置，数据库密码和 SECRET_KEY 的默认值只适合本地验证，不适合上线。

## 七、达到“可以上云”的最低标准

满足下面这些条件，再上云才稳。

### 1. 功能层

- 邮箱注册登录可用
- 手机号验证码登录可用
- 配对邀请码主链路可用
- 双方打卡和日报生成可用
- 首页、报告页、个人中心、里程碑页可用
- UI 自动回归通过

### 2. 安全层

- `SECRET_KEY` 已改为强随机值
- 生产环境关闭 `PHONE_CODE_DEBUG_RETURN`
- 不再使用默认数据库密码
- 生产环境不要保留测试账号或调试开关
- 小程序本地缓存不再存明文密码

### 3. 部署层

- Docker Compose 本地启动成功
- Nginx 转发正常
- 数据库可连通
- 上传目录可写
- API Key 已配置

### 4. 验证层

- `python test_ui.py` 通过
- `python -m py_compile` 对关键后端模块通过
- 手工完成至少一轮完整业务测试

---

## 八、上云建议顺序

推荐按下面顺序做，不容易翻车。

### 第一步：准备服务器

建议配置：

- Ubuntu 22.04
- 2C4G 起步
- 已安装 Docker 和 Docker Compose

### 第二步：上传代码

可以用 Git：

```bash
git clone 你的仓库地址
cd qinjian
```

### 第三步：配置生产环境变量

至少确保这些值不是默认值：

- `SECRET_KEY`
- `DB_PASSWORD`
- `SILICONFLOW_API_KEY`
- `PHONE_CODE_DEBUG_RETURN=false`

### 第四步：启动服务

```bash
docker compose up -d --build
docker compose ps
docker compose logs -f
```

### 第五步：绑定域名和 HTTPS

- 域名先解析到服务器
- 再由 Nginx 或云端网关处理 HTTPS
- 最后检查 `/api/health` 是否能通过域名访问

---

## 九、上云前我建议你额外处理的两件事

### 1. 清掉仓库里的硬编码敏感信息

当前仓库里仍然能搜到历史脚本中的明文服务器密码痕迹，例如：

- [auto_deploy.py](auto_deploy.py)
- [check_logs.py](check_logs.py)

这类文件如果还要继续保留，至少要先改成环境变量读取；如果已经不用，建议删除或移出仓库。

### 2. 收紧生产环境 CORS

当前后端 [backend/app/main.py](backend/app/main.py) 里 `allow_origins=["*"]` 仍是开发期写法。上线前建议改成你的正式域名，否则生产环境边界太松。

---

## 十、你现在最省事的执行顺序

如果你想最快判断“能不能上云”，按这个顺序跑：

1. 配好 `backend/.env`
2. 本地启动 PostgreSQL + uvicorn
3. 用 `python -m http.server 3000` 打开 Web
4. 手工走一遍：注册 → 配对 → 打卡 → 报告 → 里程碑 → 个人中心
5. 跑 `python test_ui.py`
6. 跑 `docker compose up -d --build`
7. 本地容器形态通过后，再上云

只要第 4、5、6 步都通过，就已经具备“可以上云试运行”的基础。
