# 亲健四阶段执行计划

## 目标

把产品路线图从“方向描述”落到“实际执行顺序”。

这份文档回答三个问题：

1. 先做什么
2. 每阶段做到什么算完成
3. 各端和后端分别要承担什么工作

## 第一阶段：核心引擎期

### 本阶段目标

把“关系画像 + 干预计划 + 效果回流”三件事做成统一底盘。

### 后端重点

- 统一关系画像字段定义
- 统一画像快照生成时机
- 把打卡、报告、任务、危机、预演、修复协议都写入关系状态层
- 为干预计划建立统一输入输出结构
- 为建议执行结果建立可追踪的反馈链

### Web / 小程序 / App 重点

- 不急着再堆很多新页面
- 先确保现有打卡、报告、任务、危机入口都能写回统一状态层
- 在关键页面给建议、执行、反馈留出产品入口

### 完成标准

- 任一重要关系事件都能沉淀到统一画像层
- 任一建议都能被追踪执行结果
- 后续高感知能力不再单独造一套数据结构

## 第二阶段：高感知产品期

### 本阶段目标

优先做最容易让用户明显感受到“这产品不一样”的能力。

### 必做三项

1. 聊天前仿真
2. 双视角叙事对齐
3. 冲突修复协议引擎

### 后端重点

- 为聊天前仿真提供统一输入、风险评分和改写建议结构
- 为双视角叙事对齐提供共同版本、错位点和桥接句结构
- 为修复协议提供分级流程、状态推进和执行记录结构

### 前端重点

- Web 先做完整主链路
- 小程序和 App 复用同一能力定义，保持体验一致
- 页面设计强调“马上能用”“立刻有帮助”，而不是堆解释文案

### 完成标准

- 用户能真实完成一次“消息预演 -> 对齐叙事 -> 打开修复协议”的完整链路
- 三项能力都能把结果回写到核心引擎层

## 第三阶段：学习增强期

### 本阶段目标

让系统从“能分析”升级到“越来越懂这对人”。

### 必做三项

- 效果学习系统
- 个体化干预偏好
- 关系数字分身

### 后端重点

- 对建议有效性做持续评估
- 给不同关系建立偏好向量或策略倾向
- 让数字分身基于真实长期数据而不是一次性 prompt

### 产品重点

- 显示“系统为什么更懂你们了”
- 显示“哪些建议对你们更有效”
- 让用户感受到推荐在变得更贴身，而不是随机换话术

### 完成标准

- 系统能根据历史效果调整建议优先级
- 同样风险情境下，不同关系能拿到不同处理建议

## 第四阶段：平台扩展期

### 本阶段目标

做第二增长曲线，而不是抢跑。

### 重点方向

- 咨询师协作模式
- 多关系图谱
- 隐私优先 AI
- 更强多模态能力

### 后端重点

- 权限和数据边界更细
- 面向咨询师的协作视图和受控授权机制
- 更强的隐私分层、摘要最小化与本地化能力
- 图片、语音、文档等多模态输入的统一处理链

### 产品重点

- C 端继续保持简单主链路
- B 端或专业协作能力做单独工作台
- 隐私表达要从“技术实现”升级到“用户可感知承诺” 

### 完成标准

- 平台能力不会破坏原有主链路
- 新增协作角色不会让普通用户操作复杂化

## 当前建议执行顺序

### 现在立刻进入开发的能力

- 聊天前仿真
- 双视角叙事对齐
- 冲突修复协议引擎

### 现在必须同步铺底的能力

- 效果学习系统的事件、反馈、评估结构

### 先不抢跑的能力

- 咨询师协作模式
- 多关系图谱
- 隐私优先 AI
- 更强本地化 / 多模态能力

## 开发协同原则

- 新功能先问：它是不是服务于四阶段路线图
- 新页面先问：它是不是复用核心引擎，而不是绕开底盘另起逻辑
- 新 AI 能力先问：它是不是能写回画像、计划和效果回流

如果不能回答这三个问题，就不应该优先开发。
