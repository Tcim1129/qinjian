# 大学生计算机设计大赛软件赛道 10 分钟演示脚本

## 演示目标

- 让评委快速确认这是一个真实可访问的 Web 软件系统
- 让评委看到完整业务闭环，而不是零散功能展示
- 让评委理解智能能力在系统中的位置是“辅助判断与辅助表达”，不是全部内容
- 让每一步都呈现输入、处理、输出和实际价值

## 演示总原则

- 固定顺序，不临场跳页面
- 一步只讲一个重点，不抢话
- 少讲模型名字，多讲系统逻辑
- 先证明系统存在，再证明系统有用，最后证明系统有技术含量

## 演示顺序总览

1. 打开公网网址
2. 完成今日记录
3. 读取当前判断中心
4. 展示消息预演
5. 展示双视角叙事对齐
6. 展示修复协议
7. 展示时间轴与证据抽屉
8. 用趋势与证据收束闭环
9. 总结系统价值

## 分钟脚本

### 0:00 - 0:40 打开公网网址

操作：

- 直接在浏览器打开项目公网地址
- 进入首页，不要提前打开太多页面

讲法：

这里先展示的是项目的真实访问入口。我们这次参赛的定位是 `软件应用与开发 > Web应用与开发`，所以第一步先证明它不是概念原型，而是可以真实访问、真实操作的 Web 系统。

### 0:40 - 1:20 进入主流程

操作：

- 进入今日记录页
- 简单说明系统接下来会从记录进入判断、干预和复盘

讲法：

这里直接进入系统主流程。我们会按照“记录、判断、预演、对齐、修复、复盘”的顺序往下走，让评委快速看清这套 Web 软件系统如何从真实输入走到可执行建议和证据回看。

### 1:20 - 2:20 完成今日记录

操作：

- 进入今日记录或打卡页面
- 录入一条准备好的示例内容

讲法：

这一环是系统的输入端。用户先把当天的关系状态、互动体验或情绪感受记录下来。对我们来说，这一步不是简单收集文本，而是把后续判断所需的上下文输入系统，为后面的关系画像、风险判断和干预建议提供依据。

### 2:20 - 3:20 当前判断中心

操作：

- 提交记录
- 打开关系简报页

讲法：

完成记录后，系统会进入当前判断中心。这里不是只有一段生成文本，而是把简报、风险状态、路径判断、趋势和下一步动作放进同一张页面里。也就是说，系统已经从“收集信息”进入“形成判断和安排行动”这一层。

### 3:20 - 4:10 当前路径与画像共享

操作：

- 指出当前路径、风险等级、当前周期等字段
- 说明这些状态会被预演、对齐、修复协议共享

讲法：

关系画像和共享状态是系统的关键中间层。它把打卡、报告、任务、风险事件等分散信息沉淀为统一关系状态。后面的消息预演、双视角对齐和修复协议，都不是各算各的，而是基于同一套上下文工作。

### 4:10 - 5:20 消息预演

操作：

- 输入一句带冲突风险的话
- 展示系统返回的感受判断、风险提示和改写建议

讲法：

消息预演体现的是“事前支持”。很多产品是在冲突发生后再做总结，我们希望在用户真正发出消息之前，先帮他看到潜在风险，给出更安全的表达方式。这一步让系统从事后分析走向事前预防，也是现场最容易感知价值的功能之一。

### 5:20 - 6:30 双视角叙事对齐

操作：

- 打开双视角叙事对齐页面
- 展示共同版本、双方关注点和错位点

讲法：

关系问题往往不是谁对谁错，而是双方叙事不一致。这个模块会把双方最近同阶段的记录进行整理，生成共同版本、重点差异和可直接开口的桥接表达。这里体现的是系统如何把抽象情绪转换成可执行沟通。

### 6:30 - 7:30 修复协议

操作：

- 打开修复协议页面
- 展示不同等级的修复步骤或当前修复建议

讲法：

如果系统判断关系风险升高，就不会只停留在“提醒你有问题”，而是进一步给出结构化修复协议。也就是说，系统从识别问题进入了行动层，开始告诉用户下一步应该怎么做。这是“软件干预能力”的关键体现。

### 7:30 - 8:40 时间轴与证据抽屉

操作：

- 打开时间轴页面
- 点击一个事件节点，展示证据抽屉或详情面板

讲法：

这一部分用来证明系统不是黑盒。我们把关键动作沉淀成关系事件流，再通过时间轴回放和证据抽屉告诉用户：系统为什么记录这个事件、这个事件影响了什么、后面为什么给出这样的建议。这里体现的是可追踪、可解释和可复盘。

### 8:40 - 9:30 用一句话收束闭环

讲法：

到这里，整个闭环就完整了：用户先输入真实关系数据，系统形成关系画像，给出风险识别和干预建议，在沟通前做预演，在冲突中做修复支持，并把这些过程持续写回系统。我们想做的不是单次回答问题，而是让软件持续参与真实关系场景。

### 9:30 - 10:00 结束语

讲法：

所以，亲健更适合放在软件应用与开发赛道。因为它的核心竞争力，不是单一算法展示，而是一套真实可访问、结构完整、具有业务闭环和智能能力嵌入的 Web 软件系统。我们的演示到这里结束，谢谢各位老师。

## 现场注意事项

- 浏览器只保留必要标签页，避免切错
- 演示账号提前准备好，避免登录失败
- 每个功能都使用预设示例，避免现场临时输入失控
- 如果网络波动，优先保住主链路：记录 -> 当前判断 -> 预演 -> 修复 -> 时间轴
- 时间轴与证据抽屉是加分项，前面时间不够时可以压缩，但不要把主链路打乱

## 高频问答口径

### 如果评委问：AI 创新在哪里？

回答：

我们的创新不在于训练了一个新模型，而在于把关系事件流、画像快照、消息预演、双视角叙事对齐和修复协议放进同一套系统，让各模块共享上下文，并且让结果可解释、可追踪、可回流。AI 在这里是对真实关系沟通的补位，而不是替代。

### 如果评委问：为什么不报人工智能应用？

回答：

因为项目当然用了 AI，但当前最成熟、最有竞争力的部分，是完整软件系统和真实演示链路。按照赛道定义，它更符合运行在 Web、网络和数据库系统之上的软件作品，所以放在软件应用与开发赛道更贴合，也更能发挥现有优势。
