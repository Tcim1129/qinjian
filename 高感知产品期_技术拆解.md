# 亲健高感知产品期技术拆解

## 目标

高感知产品期聚焦三件最容易让用户立刻感受到差异的能力：

1. 聊天前仿真
2. 双视角叙事对齐
3. 冲突修复协议引擎

这一阶段的目标不是把 AI 放到更多地方，而是让用户在真实关系场景里明显感到：这个产品在关键时刻真的能帮上忙。

## 一、聊天前仿真

### 目标

在用户把一句话发出去之前，先帮他看到风险、误读点和更安全的表达方式。

### 输入

- 用户准备发送的原句
- 当前关系画像
- 最近 7 天关键事件
- 当前风险等级

### 输出

- 对方可能的第一感受
- 升级风险评分
- 不建议直接发送的原因
- 更安全的改写版本
- 建议做法 / 避免做法

### 产品要求

- 要在 10 秒内给出可读结果
- 结果必须能直接指导用户修改表达
- 不能只给抽象心理分析，必须有可复制的话术建议

### 技术要求

- 输出结构统一 JSON 化
- 风险评分与改写建议可解释
- 结果写回关系画像与效果回流层

## 二、双视角叙事对齐

### 目标

当双方都记录了自己的版本时，系统自动整理出“共同版本”和“错位点”，帮助双方更容易展开对话。

### 输入

- 双方最近同阶段记录
- 最近报告摘要
- 当前关系画像

### 输出

- 共同版本
- A 方更在意的点
- B 方更在意的点
- 最容易误读的错位点
- 可直接使用的桥接句

### 产品要求

- 输出要适合直接展示给双方
- 不能带评判语气
- 要能帮助“开口”，不是制造新的防御

### 技术要求

- 统一对齐结果数据结构
- 支持单次生成与历史回看
- 结果写回关系画像与事件流

## 三、冲突修复协议引擎

### 目标

在关系风险升高时，不只提示问题，而是给出一套可以执行的修复步骤。

### 输入

- 当前风险等级
- 最近冲突事件
- 关系画像
- 双方近期执行历史

### 输出

- 协议等级
- 分步骤修复动作
- 每一步的目的说明
- 暂停条件 / 升级条件
- 完成后的复盘建议

### 产品要求

- 协议必须像流程，不像长文章
- 用户能逐步执行并看到当前进度
- 要区分轻度、中度、严重冲突情境

### 技术要求

- 协议结构可序列化
- 支持状态推进与中断恢复
- 执行结果写回效果回流系统

## 四、三项能力的共同要求

### 1. 必须建立在核心引擎之上

- 读取关系画像
- 写回关系画像
- 进入干预计划层
- 进入效果回流层

### 2. 必须有统一的结果结构

不能每项能力都单独定义一套前端字段和后端返回形态。

### 3. 必须可跨端复用

- Web 作为第一主链路
- 小程序与 App 后续复用同一能力定义

## 六、完成标准

高感知产品期完成，不是“加了三个页面”，而是满足：

1. 用户能真实完成一次“仿真 -> 对齐 -> 修复协议”的连续体验
2. 三项能力都能被记录、复盘和评估
3. 三项能力都不是孤立页面，而是核心引擎上的产品能力
